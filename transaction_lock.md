# その在庫、二重に売れていませんか

**在庫1個。同じ瞬間に2人が「買う」を押す。**

```text
在庫: 1
Aさん: 在庫を見る → 1個ある → 買う → 在庫 0
Bさん: 在庫を見る → 1個ある → 買う → 在庫 0   ← 同じ1個を売った
```

売れたのは **2個**。減ったのは **1個**。
これが **在庫違算**。原因は「読んでから書くまでの隙間」。

<!-- ナレーション: まず何が困るかを体感してもらいます。在庫1個を2人が同時に買うと、両方が「在庫あり」を見てしまい、1個を二重に売ってしまう。今日はこの隙間の話です。 -->

# 1. 行ロックをする目的

## 「読んで・判定して・書き戻す」の隙間を閉じる

---

# 何が起きているか（lost update）

プレーンな `SELECT` は **スナップショット読み**。
最新のコミットを待たず、自分が見た値のまま突き進む。

```text
t0  A: SELECT quantity → 10
t0  B: SELECT quantity → 10      ← 同じ 10 を読む
t1  A: 9 を書く (UPDATE = 10-1)
t2  B: 9 を書く (UPDATE = 10-1)  ← A の更新を上書き。1回ぶんが消える
```

2回減らしたのに、在庫は **9**（本来は 8）。
= **lost update（更新の消失）**。

<!-- ナレーション: 通常のSELECTはMySQLのREPEATABLE READだとスナップショットを読むので、他人のコミットが見えません。両者が同じ10を読み、片方の減算が上書きで消えます。 -->

---

# `FOR UPDATE`（悲観ロック）で直列化する

`SELECT ... FOR UPDATE` は排他ロック＋**最新コミット値**を読む。

```text
t0  A: SELECT ... FOR UPDATE → 10  （ロック取得）
t0  B: SELECT ... FOR UPDATE → 待たされる…
t1  A: 9 を書いて COMMIT（ロック解放）
t2  B: 待ち解除 → 9 を読み直す → 8 を書く
```

読み・判定・書きが1本ずつ直列に並ぶ。在庫は正しく **8**。

> ⚠️ **粒度の注意**: 単一の `UPDATE quantity = quantity - 1` だけなら
> InnoDB では原子的で違算しない。違算するのは
> **「アプリで読む → 判定する → 書き戻す」** パターン。そこにロックが要る。

<!-- ナレーション: FOR UPDATEは待たせた側に最新値を読み直させます。ここ重要なのは、単発のUPDATE文自体は原子的で安全ということ。危ないのはアプリ側に値を持ってきて判定して書き戻す形です。 -->

---

# コード: 違算する実装 vs 直した実装

```php
// ❌ BuggyStockService：ロックもトランザクションも無い
$stock = Stock::on($conn)->findOrFail($id); // 読む（スナップショット）
if ($stock->quantity < $by) throw ...;       // 判定
$stock->quantity -= $by;                     // ← 隙間で誰かが書くと上書き
$stock->save();                              // 書く
```

```php
// ✅ SafeStockService：トランザクション内で FOR UPDATE
DB::connection($conn)->transaction(function () use ($id, $by, $conn) {
    $stock = Stock::on($conn)->lockForUpdate()->findOrFail($id); // ロックして読む
    if ($stock->quantity < $by) throw ...;
    $stock->quantity -= $by;
    $stock->save();
}, attempts: 3); // デッドロック時だけ自動リトライ
```

`lockForUpdate()` → SQL の `FOR UPDATE`。**トランザクションの中でこそ効く**。

<!-- ナレーション: 実装の差はこれだけです。トランザクションで包んで、読みにlockForUpdateを付ける。ロックはトランザクションが終わると解放されるので、必ずトランザクションの中で使います。 -->

# 2. ロックの注意点

## 正しく取っているつもりで、漏れる・詰まる

---

# 注意点(1) 同種の減算は「同じロック」を取る

在庫を減らす経路が **複数** あると、片方がロックを取っても
もう片方がロック無し／別の行をロックしていれば、そこから漏れる。

```text
経路X: 注文確定   → stocks(id=5) を FOR UPDATE ✅
経路Y: 手動調整   → stocks(id=5) をロック無しで UPDATE ❌
                    → X が守っていても Y が突き抜ける
```

- **同じ資源には、同じ粒度・同じ対象のロックを、全経路で**
- 「減算は必ずこのサービスを通す」と**入口を1つに**するのが堅い

<!-- ナレーション: ロックは全員で同じ約束を守って初めて意味を持ちます。減算経路がバラバラだと、1つでもロックを取らない経路があればそこから違算します。入口を1つに絞るのが実務的な守り方です。 -->

---

# 注意点(2) テレコによるデッドロック

2つのトランザクションが、同じ2行を **逆順** にロックすると詰まる。

```text
T1: A をロック → 次に B が欲しい
T2: B をロック → 次に A が欲しい
     ↓
T1 は B 待ち、T2 は A 待ち → 循環＝デッドロック
```

- MySQL/Postgres は循環を**自動検知**し、片方をロールバック
  （MySQL: `errno 1213 / SQLSTATE 40001`、PostgreSQL: `40P01`）
- **回避策 = ロック順をそろえる**（例: 常に id の小さい順）
- 保険として `DB::transaction(fn () => ..., attempts: N)` 的なリトライもありかも？（ロジックによる）

<!-- ナレーション: テレコ、つまり逆順ロックが古典的なデッドロック原因です。直し方はシンプルで、ロックを取る順番を全トランザクションでそろえること。あとは万一に備えてattemptsでリトライを入れます。 -->

---

# 注意点(3) 空レコードによる、ギャップロック

MySQL の **REPEATABLE READ** では、範囲／非ユニーク条件の locking read は
**該当行が無くても**、スキャンした範囲（ギャップ）をロックする。

```sql
-- id=42 はまだ存在しない
SELECT * FROM stocks WHERE id = 42 FOR UPDATE;  -- ギャップロックを取る
```

```text
T1: WHERE id=42 FOR UPDATE → ギャップをロック（行は無い）
T2: WHERE id=42 FOR UPDATE → 同じギャップをロック（両立する）
T1: INSERT id=42 → T2 のギャップロックと衝突で待つ
T2: INSERT id=42 → T1 と衝突 → 循環＝デッドロック
```

「まだ無い行を予約してから作る」実装は、ここでよく詰まる。

<!-- ナレーション: これが原稿でも触れている空レコードのギャップロック。存在しない行へのFOR UPDATEでも、MySQLは範囲をロックします。2つが同じ隙間をロックして両方INSERTすると、デッドロックになります。 -->

---

# そもそもロックは「行」でなく「インデックス」に走る

InnoDB の**行ロックの実体は、インデックスレコードのロック**。

> "the row-level locks are actually **index-record locks**."
> "A record lock is a lock on **an index record**."
> — MySQL Reference Manual, InnoDB Locking

- だから**行と行の“隙間”をロックできる** ＝ ギャップロックの正体
- インデックスが無い列でも、InnoDB は隠しクラスタインデックスを使ってロックする
- **適したインデックスが無く全表スキャンになると、走査した全行がロックされ**、
  他セッションの INSERT まで止まる → 「想定外に広い範囲が詰まる」

**教訓: WHERE を効くインデックスに乗せる。ロックの範囲はインデックスで決まる。**

<!-- ナレーション: ここが土台です。InnoDBのロックは行そのものではなく、インデックスレコードに対して掛かります。だからこそ、レコードの隙間、ギャップをロックできる。逆に、適切なインデックスが無くて全表スキャンになると、走査した行が全部ロックされて、想定外に広い範囲が詰まります。ロックの範囲はインデックスで決まる、と覚えてください。 -->

---

# おまけ：ギャップロックはどこで発生する？（MySQL）

| 検索条件                                                      | 取られるロック                                |
| ------------------------------------------------------------- | --------------------------------------------- |
| **ユニークインデックス + 一意条件**（`WHERE id = 5`、行あり） | その**行だけ**（ギャップ無し）                |
| 範囲条件（`WHERE id BETWEEN 10 AND 20`）                      | 走査した**範囲にギャップ/ネクストキーロック** |
| 非ユニーク条件・該当行なし                                    | **ギャップロック**（空でも）                  |

- ネクストキーロック = 行ロック + 直前ギャップのロック
- **一意キーで1行を確実に指す**とギャップロックを避けやすい
- READ COMMITTED に下げるとギャップロックはほぼ消える（が別のリスク）

<!-- ナレーション: ギャップロックが出るのは範囲や非ユニーク条件のとき。主キーでピンポイントに1行を指すと行ロックだけで済みます。設計でロックの形をコントロールできる、というのがポイントです。 -->

# 【比較コラム】

# PostgreSQL だとどうなる？

## 「最近のPostgresなら大丈夫？」への答え

---

# MySQL と PostgreSQL は前提が違う

| 観点                                                            | MySQL (InnoDB)                 | PostgreSQL                       |
| --------------------------------------------------------------- | ------------------------------ | -------------------------------- |
| **既定の分離レベル**                                            | REPEATABLE READ                | **READ COMMITTED**               |
| **ギャップ/ネクストキーロック**                                 | あり                           | **無い**（存在しない）           |
| **未存在キーの並行INSERT**                                      | ギャップロックで抑止（詰まる） | `FOR UPDATE`では**止められない** |
| **REPEATABLE READ（RR）で同じ行を更新（書き込み競合）したとき** | 待つ（ブロック）               | **エラーで落ちる→リトライ前提**  |

> 空レコードのギャップロック地獄は **MySQL固有**。
> Postgres には**ギャップロックが無い**ので、その形のデッドロックは起きない。

<!-- ナレーション: ここが今日一番伝えたい比較です。そもそも既定の分離レベルが違う。そしてPostgresにはギャップロックという仕組み自体が無い。だから原稿で心配していた空レコードのギャップロックは、Postgresでは起きません。 -->

---

# 分離レベルが「防ぎたい」3つの異常

**① ダーティリード** — まだコミットされていない値を読む

```text
T2: 在庫 10→5 に UPDATE（未コミット）
T1: 5 を読む
T2: ROLLBACK → 実際は 10。T1 は"幻の5"を見た（存在しない値）
```

**② 反復不能読み取り** — 同じ「行」を2回読むと値が変わる

```text
T1: 在庫を読む → 10
T2: 10→8 にして COMMIT
T1: もう一度読む → 8   （同じ行なのに値が違う）
```

**③ ファントムリード** — 同じ「条件」で読むと行数が変わる

```text
T1: WHERE price<100 を数える → 5件
T2: 条件に合う行を1件 INSERT して COMMIT
T1: もう一度数える → 6件   （幽霊のように現れた行）
```

<!-- ナレーション: 分離レベルの話をする前に、そもそも何を防ぎたいのかを整理します。異常は3つ。ダーティリードは、まだコミットされていない値を読んでしまうこと。相手がロールバックすれば、存在しなかった値を読んだことになります。反復不能読み取りは、同じ行を2回読むと、他人のコミットのせいで値が変わること。ファントムは、同じ条件で数えると行の数が変わること。1件目と2件目で行が増えたり減ったりします。 -->

---

# 分離レベルの星取表

**○ = その異常が起きうる ／ × = 防ぐ（起きない）**

| 分離レベル                       | ①ダーティリード | ②反復不能読み取り | ③ファントム |
| -------------------------------- | :-------------: | :---------------: | :---------: |
| READ UNCOMMITTED                 |        ○        |         ○         |      ○      |
| **READ COMMITTED**（PG既定）     |        ×        |         ○         |      ○      |
| **REPEATABLE READ**（MySQL既定） |        ×        |         ×         |     ○ ※     |
| SERIALIZABLE                     |        ×        |         ×         |      ×      |

- ※ SQL標準では RR はファントムを許す（○）が、**MySQL・PostgreSQL の RR は実際には防ぐ（×）**
  （MySQLはネクストキーロック／ギャップロック、PostgreSQLはスナップショット分離）
- PostgreSQL に真の READ UNCOMMITTED は無く、READ COMMITTED と同じ → ダーティリードは起きない
- **③ファントムを防ぐ仕組み＝ロック。だからロックはインデックスの“隙間”に走る**

<!-- ナレーション: 星取表で整理します。マルがその異常が起きうる、バツが防ぐ、です。下にいくほど厳しくなります。READ UNCOMMITTEDは全部起きる。READ COMMITTEDはダーティリードだけ防ぐ。これがPostgresの既定です。REPEATABLE READは反復不能読み取りまで防ぐ。これがMySQLの既定です。SERIALIZABLEで全部防ぐ。注意はファントムの列で、SQL標準ではRRはファントムを許しますが、MySQLもPostgresも実際のRRではファントムを防ぎます。MySQLはネクストキーロック、Postgresはスナップショット分離で。このファントムを防ぐ仕組みがロックの正体で、だからロックはインデックスの隙間に走るわけです。 -->

---

# ファントムを防ぐ仕組み：MySQL と PostgreSQL

同じ RR でも、**ファントムの防ぎ方が根本的に違う**。

**MySQL（InnoDB）＝「入れさせない」（ロックで阻止）**

- 非ロックの `SELECT`：MVCCスナップショット
  → 「最初の読みで確定したスナップショット」を読むので、後から入った行は**見えない**
- ロック読み（`FOR UPDATE` 等）は**スナップショットを超えて最新を見にいく**（current read）
  → このままでは後から入った行が見えて**ファントムが起きうる**
  → そこで **ネクストキーロック（＝行ロック＋直前ギャップのロック）** で走査範囲を封鎖し、
  他セッションに範囲内へ **INSERT させない** ＝ ファントムを物理的に阻止

**PostgreSQL（RR ＝ スナップショット分離）＝「入っても見せない」（MVCC）**

- トランザクション先頭時点のスナップショットで**全読み取りが固定**
- 他者が後から INSERT しても、自分のスナップショットに無い → **ただ見えないだけ**
- だから **ギャップロックは不要**。INSERT を止めない（書き込み衝突だけ serialization error）

→ MySQL＝**ロックで阻止**（空レコードのギャップロック地獄・デッドロックが増えうる）
→ PostgreSQL＝**MVCCで隠す**（ブロックしないが、衝突時はエラー→リトライ）

<!-- ナレーション: あの括弧書きをちゃんと解説します。同じREPEATABLE READでも、ファントムの防ぎ方がMySQLとPostgresで根本的に違います。MySQLは入れさせない方式。普通のSELECTはMVCCのスナップショットで後から入った行が見えない。ですがFOR UPDATEのようなロック読みは、スナップショットを超えて最新を見にいきます。このままだと後から入った行が見えてしまってファントムが起きうる。だからネクストキーロック、つまり行ロックと直前のギャップのロックで走査した範囲を封鎖して、他のセッションがその範囲にINSERTできないようにします。最新を見にいくその読みが、ついでに範囲ロックを置いていくイメージで、物理的に阻止するわけです。一方Postgresは入っても見せない方式。トランザクション先頭のスナップショットで全部固定するので、後から誰かがINSERTしても、自分のスナップショットに無いから見えないだけ。だからギャップロックが要らない。挿入も止めません。この違いが、MySQLは空レコードのギャップロック地獄でデッドロックが増えうる、Postgresはブロックしないが衝突時はリトライ、という実務の差につながります。 -->

---

# 分離レベル：REPEATABLE READ と READ COMMITTED

違いは **「どの時点のスナップショットを読むか」**。

|                     | READ COMMITTED（PG既定）             | REPEATABLE READ（MySQL既定）   |
| ------------------- | ------------------------------------ | ------------------------------ |
| スナップショット    | **文ごと**に最新コミットへ更新       | **トランザクション先頭**で固定 |
| 同じ行を2回読む     | 値が変わりうる（＝反復不能読み取り） | **同じ値**が返る               |
| 同じ条件を2回数える | 行数が変わりうる（＝ファントム）     | 変わらない（実質防ぐ）         |

- RR は「読み始めた世界」で一貫する反面、**他人のコミットが見えない**
  → 在庫を plain SELECT で見直しても**古い値のまま**。`FOR UPDATE` は最新を読む
- **どちらの分離レベルでも、read→判定→write の lost update は防げない**
  → 直列化するのは分離レベルではなく **ロック（FOR UPDATE）**

<!-- ナレーション: 分離レベルの違いは、どの時点のスナップショットを読むか、に尽きます。READ COMMITTEDは文ごとに最新コミットを見にいくので、同じSELECTでも値が変わりうる。REPEATABLE READはトランザクションの先頭で固定されるので、何度読んでも同じ値。ここで大事なのは、どちらの分離レベルを選んでも、読んで判定して書き戻すlost updateは防げないということです。直列化してくれるのは分離レベルではなく、FOR UPDATEのロックです。 -->

---

# 補足：RR でも lost update は防げない

分離レベルを RR に上げても、`FOR UPDATE` 無しの read→判定→write は消失する。

```text
在庫 = 10

t0  T1: SELECT quantity → 10        （RR：先頭でスナップショット固定）
t0  T2: SELECT quantity → 10
t1  T2: 10-1=9 を UPDATE → COMMIT
t2  T1: SELECT quantity → 10 ！      ← plain SELECT はスナップショットのまま古い 10
t2  T1: 10-1=9 を UPDATE → COMMIT    ← T2 の 9 を上書き。1回ぶんが消える
```

- **2回引いたのに在庫 9**（本来 8）＝ lost update
- RR はむしろ**再読しても他人のコミットが見えず 10 のまま** → 古い値で上書きしがち
- 分離レベルが決めるのは「**読みが何を見るか**」だけ。「**読み〜書きの隙間**」は塞げない
  → 塞ぐのは `FOR UPDATE`（最新を読み直し、他人を待たせて直列化）

<!-- ナレーション: 具体例で確かめます。分離レベルをMySQL既定のREPEATABLE READに上げても、FOR UPDATE無しなら消失は防げません。T1とT2が同じ10を読む。T2が先に9を書いてコミット。ここでT1がもう一度plain SELECTしても、REPEATABLE READはスナップショットが先頭で固定されているので、まだ10に見えます。他人のコミットが見えないんですね。だからT1は古い10を信じて9を書き、T2の更新を上書きしてしまう。2回引いたのに在庫は9、本来は8です。ポイントは、分離レベルが制御するのは読みが何を見るかだけで、読んでから書くまでの隙間は塞げないということ。その隙間を塞ぐのがFOR UPDATEで、最新を読み直して他人を待たせて直列化します。 -->

---

# ただし「根っこの問題」は Postgres でも残る

Postgres の `SELECT ... FOR UPDATE` は **存在する行しかロックしない**。
= まだ無いキーの並行INSERTは**止められない**。

- **READ COMMITTED**: 競合行はブロック後、**最新コミット版でWHEREを再評価**して
  ロック・返却（EvalPlanQual）。スナップショットとズレることがある
- **REPEATABLE READ**: 他者が更新済みの行を触ると
  `ERROR: could not serialize access due to concurrent update`
  → **アプリでキャッチして丸ごとリトライ**する設計が要る

**未存在キーの競合を守る手段** →
ユニーク制約 / `INSERT ... ON CONFLICT` / SERIALIZABLE / advisory lock

<!-- ナレーション: ただしギャップロックが無いということは、まだ無い行の競合はFOR UPDATEでは守れないということでもあります。Postgresではユニーク制約やON CONFLICT、あるいはSERIALIZABLEで守ります。RRは待つのではなくエラーで落ちるので、リトライ前提の作りになります。 -->

# 注意点(4) 辛くなったらキャッシュ層のロック？

DBロックが重い／ホットな1行に集中する、となると
Redis 等の**分散ロック**を前段に置きたくなる。でも——

- **Redlock は「効率化」であって「正しさの担保」ではない**（論争あり）
  - Kleppmann: ネットワーク遅延・GCポーズ・時計ずれで壊れうる／
    fencing token が無いと遅延した旧ロック保持者の書き込みを弾けない
  - antirez: 反論あり。**決着していない**
- 在庫のような**お金に近い正しさ**では、真実の源は **DBのトランザクション/制約**
- Redisロックは「DBの前段のスループット最適化」と割り切る

<!-- ナレーション: DBロックが辛くなるとキャッシュ層のロックを検討したくなりますが、分散ロックは正しさの担保としては論争があります。在庫のように正しさが重要なものは、DBのトランザクションと制約を真実の源に据えて、Redisは効率化にとどめるのが無難です。 -->

# 3. ロックをしない解決法

## アトミックな条件付きUPDATE

---

# 在庫減算は UPDATE 一発に畳める

「読む→判定→書き戻す」の代わりに、**1文で条件付きに引く**。

```sql
UPDATE stocks
SET quantity = quantity - 1
WHERE sku = 'A-001' AND quantity >= 1;   -- ← ガード（在庫がある時だけ）
-- 在庫不足なら「0行マッチ」で終わる（例外は飛ばない）
```

- **1文なので read→判定→write の隙間が無い** → lost update しない
- `FOR UPDATE` もアプリ側ロックも不要（MySQL・PostgreSQL とも既定で安全）

<!-- ナレーション: 在庫を1つ減らすだけなら、読んで判定して書き戻す代わりに、条件付きのUPDATE一発に畳めます。WHERE quantity >= 1 のガードで、在庫がある時だけ引く。1文なので読んで判定して書き戻す隙間が無く、lost update が起きません。FOR UPDATE もアプリ側のロックも要らず、MySQLでもPostgresでも既定で安全です。この書き方の弱点は、このあとのスライドで扱います。 -->

---

# 落とし穴：ガードを外すとマイナス在庫

**アトミック更新の弱点は「マイナスを作れてしまう」こと。**

```sql
-- ❌ ガード無し：在庫0でも引けてしまう → 売りすぎ（quantity がマイナス）
UPDATE stocks SET quantity = quantity - 1 WHERE sku = 'A-001';
```

守りは2段構え:

- **必ず `WHERE quantity >= 1` を付ける** … そもそもマイナスにしない＋`affected=0`で検知
- 保険に **`UNSIGNED` 列** か **`CHECK (quantity >= 0)`**（MySQL 8.0.16+／PG）でDBからも弾く

そして——**判定が複雑で1文に畳めない**
（複数行を読んで計算し、複数箇所を書く）ときは、素直に
**トランザクション + `FOR UPDATE`** に戻す。アトミック更新は万能ではない。

<!-- ナレーション: ただしアトミック更新には弱点があります。ガードを外すと、在庫ゼロでも引けてしまって在庫がマイナス、つまり売りすぎになります。だから必ずWHERE quantity >= 1のガードを付ける。加えて、unsigned列やCHECK制約でDB側からも弾いておくと保険になります。そして判定が複雑で1文に畳めないときは、無理せずトランザクションとFOR UPDATEに戻します。アトミック更新は万能ではありません。 -->

---

# もう一つの弱点：在庫不足でも例外は飛ばない

ガード（`WHERE quantity >= 1`）を付けても、**在庫不足は「エラー」にならない**。

```sql
UPDATE stocks SET quantity = quantity - 1
WHERE sku = 'A-001' AND quantity >= 1;
-- 在庫不足 → 「0行にマッチした成功」。例外は飛ばない
```

- DB 的には**正常終了**。呼び出し側は失敗に気づけない
- **影響行数(affected rows)を自分で見て、0 なら在庫不足として自前で throw** する
- チェックを忘れると “売り切れなのに成功” ＝ **サイレント失敗**

> `FOR UPDATE` + `if` 方式なら **`if` で明示的に投げられる**のが対照的な強み。
> なお `affected=0` は「在庫不足」と「SKU不在」を区別しない → 分けたいなら別途存在確認を。

<!-- ナレーション: マイナス在庫を防いだうえで、もう一つの弱点があります。ガードを付けても、在庫不足のときに例外が飛んでくれません。ガードのWHEREで弾かれても、DB的にはゼロ行にマッチした成功として正常終了するだけ。だからアプリ側で影響行数を自分でチェックして、ゼロなら在庫不足として自分で例外を投げる必要があります。このチェックを忘れると、売り切れなのに成功扱いになるサイレントな失敗になります。読んで判定してif文で明示的に投げるFOR UPDATEのやり方なら、ここは明示的に投げられるので対照的です。ちなみに影響行数ゼロは、在庫不足なのか、そもそもSKUが存在しないのかを区別しないので、分けたいなら別途存在確認が要ります。 -->

# まとめ：設計原則

1. **読み・判定・書きの“隙間”をロックで閉じる**
   トランザクション + `lockForUpdate()`。単発UPDATEなら不要
2. **減算経路を1つに**（全経路で同じロックを取る）
3. **ロック順をそろえる**（テレコ＝逆順でデッドロック、attemptsで保険）
4. **ギャップロックは MySQL 固有**
   Postgres は既定 READ COMMITTED・ギャップロック無し・RRはエラー→リトライ
5. **単純な減算はアトミックUPDATEで**
   `WHERE quantity >= n` のガード必須（外すとマイナス在庫）

<!-- ナレーション: まとめです。基本はトランザクションとFOR UPDATEで隙間を閉じる。経路を1つに、ロック順をそろえる。ギャップロックはMySQL固有でPostgresは別モデル。そして単純な減算はアトミックなUPDATEで、ただしガードを忘れずに。この5つを持ち帰ってください。 -->

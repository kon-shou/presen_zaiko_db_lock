# その在庫、二重に売れていませんか

**在庫1個。同じ瞬間に2人が「買う」を押す。**

```text
在庫: 1
Aさん: 在庫を見る → 1個ある → 買う → 在庫 0
Bさん: 在庫を見る → 1個ある → 買う → 在庫 0   ← 同じ1個を売った
```

売れたのは **2個**。減ったのは **1個**。
これが **在庫違算**。原因は「読んでから書くまでの隙間」。

<!-- ナレーション: まず何が困るかを体感してもらいます。在庫1個を2人が同時に買うと、両方が「在庫あり」を見てしまい、1個を二重に売ってしまう。今日はこの隙間の話です。 -->

# 1. 行ロックをする目的

## 「読んで・判定して・書き戻す」の隙間を閉じる

---

# 何が起きているか（lost update）

プレーンな `SELECT` は **スナップショット読み**。
最新のコミットを待たず、自分が見た値のまま突き進む。

```text
t0  A: SELECT quantity → 10
t0  B: SELECT quantity → 10      ← 同じ 10 を読む
t1  A: 9 を書く (UPDATE = 10-1)
t2  B: 9 を書く (UPDATE = 10-1)  ← A の更新を上書き。1回ぶんが消える
```

2回減らしたのに、在庫は **9**（本来は 8）。
= **lost update（更新の消失）**。

<!-- ナレーション: 通常のSELECTはMySQLのREPEATABLE READだとスナップショットを読むので、他人のコミットが見えません。両者が同じ10を読み、片方の減算が上書きで消えます。 -->

---

# `FOR UPDATE`（悲観ロック）で直列化する

`SELECT ... FOR UPDATE` は排他ロック＋**最新コミット値**を読む。

```text
t0  A: SELECT ... FOR UPDATE → 10  （ロック取得）
t0  B: SELECT ... FOR UPDATE → 待たされる…
t1  A: 9 を書いて COMMIT（ロック解放）
t2  B: 待ち解除 → 9 を読み直す → 8 を書く
```

読み・判定・書きが1本ずつ直列に並ぶ。在庫は正しく **8**。

<!-- ナレーション: FOR UPDATEは待たせた側に最新値を読み直させます。ここ重要なのは、単発のUPDATE文自体は原子的で安全ということ。危ないのはアプリ側に値を持ってきて判定して書き戻す形です。 -->

---

# コード: 違算する実装 vs 直した実装

```php
// ❌ BuggyStockService：ロックもトランザクションも無い
$stock = Stock::on($conn)->findOrFail($id); // 読む（スナップショット）
if ($stock->quantity < $by) throw ...;       // 判定
$stock->quantity -= $by;                     // ← 隙間で誰かが書くと上書き
$stock->save();                              // 書く
```

```php
// ✅ SafeStockService：トランザクション内で FOR UPDATE
DB::connection($conn)->transaction(function () use ($id, $by, $conn) {
    $stock = Stock::on($conn)->lockForUpdate()->findOrFail($id); // ロックして読む
    if ($stock->quantity < $by) throw ...;
    $stock->quantity -= $by;
    $stock->save();
}, attempts: 3); // デッドロック時だけ自動リトライ
```

`lockForUpdate()` → SQL の `FOR UPDATE`。**トランザクションの中でこそ効く**。

<!-- ナレーション: 実装の差はこれだけです。トランザクションで包んで、読みにlockForUpdateを付ける。ロックはトランザクションが終わると解放されるので、必ずトランザクションの中で使います。 -->

# 2. ロックの注意点

## 正しく取っているつもりで、漏れる・詰まる

---

# 注意点(1) 同種の減算は「同じロック」を取る

在庫を減らす経路が **複数** あると、片方がロックを取っても
もう片方がロック無し／別の行をロックしていれば、そこから漏れる。

```text
経路X: 注文確定   → stocks(id=5) を FOR UPDATE ✅
経路Y: 手動調整   → stocks(id=5) をロック無しで UPDATE ❌
                    → X が守っていても Y が突き抜ける
```

- **同じ資源には、同じ粒度・同じ対象のロックを、全経路で**
- 「減算は必ずこのサービスを通す」と**入口を1つに**するのが堅い

<!-- ナレーション: ロックは全員で同じ約束を守って初めて意味を持ちます。減算経路がバラバラだと、1つでもロックを取らない経路があればそこから違算します。入口を1つに絞るのが実務的な守り方です。 -->

---

# 注意点(2) テレコによるデッドロック

2つのトランザクションが、同じ2行を **逆順** にロックすると詰まる。

```text
T1: A をロック → 次に B が欲しい
T2: B をロック → 次に A が欲しい
     ↓
T1 は B 待ち、T2 は A 待ち → 循環＝デッドロック
```

- MySQL/Postgres は循環を**自動検知**し、片方をロールバック
  （MySQL: `errno 1213 / SQLSTATE 40001`、PostgreSQL: `40P01`）
- **回避策 = ロック順をそろえる**（例: 常に id の小さい順）
- 保険として `DB::transaction(fn () => ..., attempts: N)` 的なリトライもありかも？（ロジックによる）
- テレコを防ぐために、トランザクションのネスト（=save point）は、できるだけ避けたい、てのが個人的な肌感。

<!-- ナレーション: テレコ、つまり逆順ロックが古典的なデッドロック原因です。直し方はシンプルで、ロックを取る順番を全トランザクションでそろえること。あとは万一に備えてattemptsでリトライを入れます。もう一つ、テレコを防ぐという意味では、トランザクションのネスト、つまり save point もできるだけ避けたい、というのが私の肌感です。この具体策は次のスライドで話します。 -->

---

# 注意点(2)補足 ネストを避ける具体策

```php
class StockService
{
    // $manageTransaction:「Tx＋ロック」のセットをこのメソッドが管理するか
    public function decrement(int $id, bool $manageTransaction = true): void
    {
        if ($manageTransaction) {
            // 自分で「Tx＋ロック」のセットを張る
            DB::transaction(function () use ($id) {
                $stock = Stock::lockForUpdate()->findOrFail($id);
                $stock->decrement('quantity');
            });
            return;
        }

        // 親が「Tx＋ロック」のセットを持っている前提。自分ではどちらも張らない
        if (DB::transactionLevel() === 0) {
            throw new LogicException('親のトランザクション内で呼ぶこと');
        }
        Stock::findOrFail($id)->decrement('quantity');
    }
}

// 1. 親が「Tx＋ロック」のセットを張る → 子は manageTransaction: false
DB::transaction(function () use ($service) {
    Stock::lockForUpdate()->findOrFail(5); // 親がロック。セットは親スコープで成立

    $service->decrement(5, manageTransaction: false); // 子はTxもロックも張らない
});

// 2. 親トランザクションが無い単体呼び出し → デフォルト true で自分でセットを張る
$service->decrement(5);
```

<!-- ナレーション: ネストについて補足です。トランザクションのネスト、つまり save point は、できるだけ避けたいというのが私の肌感です。ネストが深いと、どこでロックを取ってどこで解放されるかが追いにくくなり、テレコも起きやすくなる。ここで大事なのは、トランザクションとロックはセットだということ。このフラグは、そのセットをどちらのスコープが持つかの切り替えです。使い分けは2パターンの対比で覚えてください。1つ目、親がトランザクションを張り、その中でロックを取ってから、子に false を渡して呼ぶ。このとき子はトランザクションもロックも張らず、実処理だけ行います。セットは親のスコープで成立している。2つ目、親トランザクションが無い単体呼び出し。このときはデフォルトの true のままで、自分でトランザクションとロックのセットを張る。なお false は「親がロック済み」という規約が前提です。transactionLevel のチェックはトランザクションの外で呼ばれたことしか検査できない、最低限の保険だと思ってください。 -->

# 注意点(3) 空レコードによる、ギャップロック

MySQL の **REPEATABLE READ** では、範囲／非ユニーク条件の locking read は
**該当行が無くても**、スキャンした範囲（ギャップ）をロックする。

```sql
-- id=42 はまだ存在しない
SELECT * FROM stocks WHERE id = 42 FOR UPDATE;  -- ギャップロックを取る
```

```text
T1: WHERE id=42 FOR UPDATE → ギャップをロック（行は無い）
T2: WHERE id=42 FOR UPDATE → 同じギャップをロック（両立する）
T1: INSERT id=42 → T2 のギャップロックと衝突で待つ
T2: INSERT id=42 → T1 と衝突 → 循環＝デッドロック
```

「まだ無い行を予約してから作る」実装は、ここでよく詰まる。

**対策：行が無ければ例外**（無い行を `FOR UPDATE` しに行かない）

```php
$stock = Stock::on($conn)->lockForUpdate()->find($id); // 一意キーで1行を指す
if ($stock === null) {
    throw new StockNotFoundException($id); // 予約INSERTに進ませない
}
```

<!-- ナレーション: これが原稿でも触れている空レコードのギャップロック。存在しない行へのFOR UPDATEでも、MySQLは範囲をロックします。2つが同じ隙間をロックして両方INSERTすると、デッドロックになります。対策はシンプルで、在庫行は事前に存在させておき、FOR UPDATE で1行も取れなければ、その時点で在庫行が無いとみなして例外を投げること。取れなかったら例外、という私の実務での対処がこれです。無い行を予約してから作る、という危険なパターンにそもそも踏み込ませないのがポイント。主キーで1行を確実に指せば行ロックだけで済み、ギャップロックも避けられます。どうしても無ければ作りたい場合は、FOR UPDATE ではなく、ユニーク制約で重複INSERTを許さない形、後半のセクション3の思想に寄せます。 -->

# 3. 行ロックとは別の解決法

---

# アトミック更新 = 在庫減算は UPDATE 一発に畳める

「読む→判定→書き戻す」の代わりに、**1文で条件付きに引く**。

```sql
UPDATE stocks
SET quantity = quantity - 1
WHERE sku = 'A-001' AND quantity >= 1;   -- ← ガード（在庫がある時だけ）
-- 在庫不足なら「0行マッチ」で終わる（例外は飛ばない）
```

- **1文なので read→判定→write の隙間が無い** → lost update しない
- `FOR UPDATE` もアプリ側ロックも不要（MySQL・PostgreSQL とも既定で安全）

<!-- ナレーション: 在庫を1つ減らすだけなら、読んで判定して書き戻す代わりに、条件付きのUPDATE一発に畳めます。WHERE quantity >= 1 のガードで、在庫がある時だけ引く。1文なので読んで判定して書き戻す隙間が無く、lost update が起きません。FOR UPDATE もアプリ側のロックも要らず、MySQLでもPostgresでも既定で安全です。この書き方の弱点は、このあとのスライドで扱います。 -->

---

# アトミック更新の落とし穴１：ガードを外すとマイナス在庫

**アトミック更新の弱点は「マイナスを作れてしまう」こと。**

```sql
-- ❌ ガード無し：在庫0でも引けてしまう → 売りすぎ（quantity がマイナス）
UPDATE stocks SET quantity = quantity - 1 WHERE sku = 'A-001';
```

守りは2段構え:

- **必ず `WHERE quantity >= 1` を付ける** … そもそもマイナスにしない＋`affected=0`で検知
- 保険に **`UNSIGNED` 列** か **`CHECK (quantity >= 0)`**（MySQL 8.0.16+／PG）でDBからも弾く

そして——**判定が複雑で1文に畳めない**
（複数行を読んで計算し、複数箇所を書く）ときは、素直に
**トランザクション + `FOR UPDATE`** に戻す。アトミック更新は万能ではない。

<!-- ナレーション: ただしアトミック更新には弱点があります。ガードを外すと、在庫ゼロでも引けてしまって在庫がマイナス、つまり売りすぎになります。だから必ずWHERE quantity >= 1のガードを付ける。加えて、unsigned列やCHECK制約でDB側からも弾いておくと保険になります。そして判定が複雑で1文に畳めないときは、無理せずトランザクションとFOR UPDATEに戻します。アトミック更新は万能ではありません。 -->

---

# アトミック更新の落とし穴２：ガード付けたら今度は例外が飛ばない

ガード（`WHERE quantity >= 1`）を付けても、**在庫不足は「エラー」にならない**。

```sql
UPDATE stocks SET quantity = quantity - 1
WHERE sku = 'A-001' AND quantity >= 1;
-- 在庫不足 → 「0行にマッチした成功」。例外は飛ばない
```

- DB 的には**正常終了**。呼び出し側は失敗に気づけない
- **影響行数(affected rows)を自分で見て、0 なら在庫不足として自前で throw** する
- チェックを忘れると “売り切れなのに成功” ＝ **サイレント失敗**

> `FOR UPDATE` + `if` 方式なら **`if` で明示的に投げられる**のが対照的な強み。
> なお `affected=0` は「在庫不足」と「SKU不在」を区別しない → 分けたいなら別途存在確認を。

<!-- ナレーション: マイナス在庫を防いだうえで、もう一つの弱点があります。ガードを付けても、在庫不足のときに例外が飛んでくれません。ガードのWHEREで弾かれても、DB的にはゼロ行にマッチした成功として正常終了するだけ。だからアプリ側で影響行数を自分でチェックして、ゼロなら在庫不足として自分で例外を投げる必要があります。このチェックを忘れると、売り切れなのに成功扱いになるサイレントな失敗になります。読んで判定してif文で明示的に投げるFOR UPDATEのやり方なら、ここは明示的に投げられるので対照的です。ちなみに影響行数ゼロは、在庫不足なのか、そもそもSKUが存在しないのかを区別しないので、分けたいなら別途存在確認が要ります。 -->

---

# UPSERT（ON DUPLICATE / ON CONFLICT）

注意点(3)の「無い行を予約してから作る」は、**upsert 1文**に畳めば
`FOR UPDATE` もギャップロックも要らない。

```sql
-- MySQL: 在庫行が無ければ作り、あれば足す（1文・原子的）
INSERT INTO stocks (sku, quantity) VALUES ('A-001', 10)
ON DUPLICATE KEY UPDATE quantity = quantity + VALUES(quantity);

-- PostgreSQL: 同じことを ON CONFLICT で
INSERT INTO stocks (sku, quantity) VALUES ('A-001', 10)
ON CONFLICT (sku) DO UPDATE SET quantity = stocks.quantity + EXCLUDED.quantity;
```

- 前提は **`sku` にユニーク制約**。重複INSERTをDBが弾く＝並行でも1行に収束
- 「存在しないキーを `FOR UPDATE`」→ ギャップロック地獄、を**構造的に回避**
- ただし… PKの意図しない増加（UPDATE落ちでも AUTO_INCREMENT を消費＝欠番）や、
  謎のデッドロック（複数ユニークキー絡み）を踏んだ経験より、**いい思い出はない…**

<!-- ナレーション: 注意点3で困っていた「無ければ作る」は、upsert一文に畳むのが本命の解決策です。MySQLならON DUPLICATE KEY UPDATE、PostgresならON CONFLICT。前提はskuにユニーク制約を張ること。これで重複INSERTをDBが弾いてくれるので、存在しないキーをFOR UPDATEしてギャップロックで詰まる、という構造そのものを避けられます。とはいえ正直に言うと、私自身はupsertでいい思い出があまりなくて。AUTO_INCREMENTのPKが失敗INSERTのぶんまで飛んで意図せず増えたり、複数ユニークキーが絡んで謎のデッドロックを踏んだり。理屈はきれいなんですが、使うなら挙動をよく確かめてから、というのが実感です。 -->

---

# 分散ロック：Redis と アドバイザリロック

DBロックが重い／ホットな1行に集中 → **名前に対して取る論理ロック**という選択肢。

**Redis（DBの外）** — Laravel なら `Cache::lock()`

```php
$lock = Cache::lock('stock:A-001', 10); // キー, TTL 10秒
if ($lock->get()) {
    try { /* 在庫処理 */ } finally { $lock->release(); }
}
```

- ただし **TTLで区切る**ため、フリーズ中（GCポーズ等）に失効→二重書きの穴（Redlock論争）＝**効率化どまり**

**アドバイザリロック（DBの中）** — MySQL `GET_LOCK()` ／ PG `pg_advisory_xact_lock()`

- 行が無くても「名前」でロックできる。DBの中なので Redis より整合が取りやすい
- **TTLが無くトランザクションに紐づく**→フリーズしても失効せず、Redisの穴が無い（※Laravel標準の文法は無し）

<!-- ナレーション: DBロックが辛い、ホットな1行に集中する、というときは、名前に対して取る論理ロックが選択肢になります。RedisはDBの外でキーにロックする方式で手軽です。ただなぜ論争があるかというと、Redisロックは、プロセスが死んだときにロックが残りっぱなしにならないよう、TTL、つまり有効期限で時間区切りするしかありません。ここで、GCポーズやネットワーク遅延で処理がTTLを超えてフリーズすると、Redisは「もう死んだ」とみなしてロックを外し、別のプロセスが入ってくる。フリーズから復帰した側は、自分のロックが切れたことに気づかないまま書いてしまうので、二重書き、つまり在庫違算になります。Redlockは効率化であって正しさの担保ではない、という論争がこれです。一方アドバイザリロックはDBの中で、ロックがトランザクションに紐づきます。TTLで時間区切りしないので、どれだけフリーズしても、トランザクションが終わるまでロックは外れない。だからRedisのこの穴が原理的に無いんです。行が無くても名前でロックできるのも利点です。ただしどちらも助言的で、全経路が同じキーで取る約束が要ります。そして大前提として、行ロックで足りるなら行ロックでいい。分散ロックはあくまで負荷対策で持ち出すもので、在庫のように正しさが命のものは、真実の源をDBのトランザクションと制約に置くのが基本です。 -->

---

# まとめ：設計原則

**基本：読み・判定・書きの“隙間”をロックで閉じる**
トランザクション + `lockForUpdate()`。単発の条件付きUPDATEなら不要

1. **行ロックの目的** … lost update を防ぐ＝並行実行を直列化する
2. **行ロックの注意点**（正しく取っているつもりで漏れる・詰まる）
   1. 減算経路を1つに … 全経路で同じ粒度・対象のロックを取る
   2. ロック順をそろえる … テレコ＝逆順でデッドロック
   3. 無い行を `FOR UPDATE` しない … 空レコードのギャップロック回避（無ければ例外）
3. **行ロック以外の選択肢**を用途で使い分ける（真実の源は常にDB）
   - アトミックUPDATE（`WHERE quantity >= n` のガード。外すとマイナス在庫／`affected=0` を自前で throw）
   - UPSERT（ユニーク制約でギャップロック回避。AUTO INCREMENTに注意）
   - 分散ロック（Redis／advisory）
